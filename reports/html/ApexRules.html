Basic

Priority - P1

Avoid HardCoded URLs

  • Especially, SalesForce server instances.
  • If the server instance address changes, there is every possibility of code being broken due to wrong URL.

Example:

    Server server = new Server();

   //Bad Practice
    server. accessURL ='.ns1.com';

Priority - P1

Hard Coding Fields

  • Hard Coding of field names and values should be avoided.
  • Instead of directly hard coding values , it is Best practice to declare them as final static variables or as Enum and use them across all the classes.
  • If single hard coded string which is used only in Apex then use constant variable.
  • If group of hard coded string which are used only in Apex then use Enum.
  • If a hard coded string is available which will be shown in UI then use Custom Label.
  • If a hard coded string is available which will vary in environment to environment then use Custom Settings.

Example:

public class HardCodingFields {

   //Bad Practice
   public void HardCodingFieldsMethod(){
      Account acc = new Account();
      acc.Name ="Acme"
   }
   //Best Practice
   public void HardCodingFieldsMethod {

      //1. example to use of Constant Class for hard coded strings.
      Account acc = new Account();
      acc.Name=UTIL_Constant.ACC_NAME;

      //2. example to use of Custom Labels for hard coded strings.
      ApexPages.addMessage(new ApexPages.Message(ApexPages.Severity.ERROR, LABEL.CL_0000));

       //3. example to use of ENUM for hard coded strings.
      String errorStatus = ErrorStatus.DMLEXCEPTION;
   }
}

Exceptions:


 - Only String literals to be considered.
 - String literals in System.debug statements to be ignored.
 - String literals in System.assert statements to be ignored.
 - Static Final variables will be ignored.

Precede Null Check

  • All Null checks should precede the IsEmpty check.

         //Do some activity

         //Bad Practice
         If (accountName!= ''){

             //Do some activity
         }

         //Best Practice.
         If (!accountName.isNULL() && accountName!= '') {

             //Do some activity

         }

         //Best Practice.
         If (accountName! = NULL && accountName!= ''){

             //Do some activity

         }

         //Best Practice
         If (!accountName.isEmpty()){

             //Do some activity

         }
     }
}

Priority - P3

Multiple Exit points

  • Multiple Exit points from a Single Method (eg: Multiple return statements in a single method) should be avoided.
  • Instead of returning multiple exit points, it is Best practice to assign the value to a variable and return that variable only once.

Example:

public class MultipleExitPoints {

  public Boolean MultipleExitPointsMethod(Integer accountBalance){

    Boolean notEnoughBalance =false;

    //Bad practice - Multiple return statements
    if (accountBalance <1000) {
        return true;
    }
    else {
        return false;
    }

    //Best Practice - Single return statement.
    If (accountBalance <1000) {
        notEnoughBalance=true;
    }
    else {
        notEnoughBalance=false;
    }
    return notEnoughBalance;
  }
}

Governor Limit Priority - P1

SOQL in Loops

Avoid SOQL or DML statements in loops.Rewrite code that executes queries inside for loops by moving the query outside of the loop and extending it to query all necessary data.
By moving queries outside of for loops, your code will run faster, and is less likely to exceed governor limits. A common mistake is that queries are placed inside a "for" loop. There is a governor limit that enforces a maximum number of SOQL queries. When queries are placed inside a "for" loop, a query is executed on each iteration and governor limit is easily reached. Instead, move the SOQL query outside of the for loop and retrieve all the necessary data in a single query.

Priority - P2

Empty catch blocks

  • Avoid having empty catch blocks.
  • Empty Catch Block finds instances where an exception is caught,but nothing is done.
  • In most circumstances, this swallows an exception which should either be acted on or reported.

Example:

public class EmptyCatchBlock {

    public void EmptyCatchBlockMethod(){

       try {
             // Create a list with two references to the same Object element.
             Account a = new Account();
             Account[] accs = new Account[]{a, a};

             // Attempt to insert it.... Will throw exception.
             insert accs;
       }

       //Bad practice - Doing nothing after the exception is thrown.
       catch (ListException e) {
       }

       //Best Practice -Doing something with the exception caught.
        catch (ListException e) {
           throw new ApplicationException(e)
       }
    }
}

Exceptions:

- System.debug statements should be ignored while checking the empty catch blocks.
